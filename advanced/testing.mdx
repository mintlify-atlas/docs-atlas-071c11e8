Electron does not ship its own testing framework, but it integrates with the broader JavaScript testing ecosystem. This page covers the main approaches: end-to-end testing with WebdriverIO or Playwright, and headless testing on Linux CI systems.

## End-to-end testing

End-to-end tests launch your real application and interact with it the same way a user would—clicking buttons, checking text, taking screenshots.

<Tabs>
  <Tab title="WebdriverIO">
    [WebdriverIO](https://webdriver.io/) (WDIO) is a Node.js test framework built on the WebDriver protocol. Its official Electron service handles launching and connecting to your app automatically.

    <Steps>
      <Step title="Initialize WebdriverIO">
        Run the configuration wizard from your project root. When asked what type of testing you want to do, select **Desktop Testing - of Electron Applications**.

        ```sh
        npm init wdio@latest ./
        ```

        This installs all required packages and generates a `wdio.conf.js` file.
      </Step>

      <Step title="Review the generated configuration">
        The wizard produces a configuration that includes the Electron service and sets `browserName` to `electron`:

        ```js wdio.conf.js
        export const config = {
          services: ['electron'],
          capabilities: [{
            browserName: 'electron',
            'wdio:electronServiceOptions': {
              // WebdriverIO auto-detects Electron Forge and electron-builder apps.
              // For other setups, specify the binary path explicitly:
              // appBinaryPath: './path/to/bundled/application.exe',
              appArgs: ['foo', 'bar=baz']
            }
          }]
        }
        ```
      </Step>

      <Step title="Write your first test">
        Use the [WebdriverIO API](https://webdriver.io/docs/api) to interact with your app's UI, and the `browser.electron.execute` helper to call Electron APIs directly from your tests:

        ```js keyboard.spec.js
        import { browser, $, expect } from '@wdio/globals'

        describe('keyboard input', () => {
          it('detects keyboard input', async () => {
            await browser.keys(['y', 'o'])
            await expect($('keypress-count')).toHaveText('YO')
          })
        })
        ```

        You can also drive Electron APIs—for example, triggering a dialog:

        ```js dialog.spec.js
        import { browser } from '@wdio/globals'

        describe('message dialog', () => {
          it('can be triggered from a test', async () => {
            await browser.electron.execute(
              (electron, a, b, c) => {
                const win = electron.BrowserWindow.getFocusedWindow()
                electron.dialog.showMessageBox(win, {
                  message: 'Hello from a test!',
                  detail: `${a} + ${b} + ${c} = ${a + b + c}`
                })
              },
              1, 2, 3
            )
          })
        })
        ```
      </Step>

      <Step title="Run your tests">
        ```sh
        npx wdio run wdio.conf.js
        ```

        WebdriverIO launches your app before the suite runs and shuts it down when the suite finishes.
      </Step>
    </Steps>

    <Note>
      If you already use WebdriverIO, update your dependencies and validate your configuration against the [official Electron setup guide](https://webdriver.io/docs/desktop-testing/electron#configuration).
    </Note>
  </Tab>

  <Tab title="Playwright">
    [Microsoft Playwright](https://playwright.dev) is an end-to-end testing framework with experimental Electron support via the [Chrome DevTools Protocol](https://chromedevtools.github.io/devtools-protocol/).

    <Steps>
      <Step title="Install Playwright">
        ```sh
        npm install --save-dev @playwright/test
        ```
      </Step>

      <Step title="Launch your app in tests">
        Use `electron.launch` and pass the path to your main process entry point:

        ```js example.spec.js
        import { test, _electron as electron } from '@playwright/test'

        test('launch app', async () => {
          const electronApp = await electron.launch({ args: ['.'] })
          await electronApp.close()
        })
        ```
      </Step>

      <Step title="Access main process APIs">
        `electronApp.evaluate` runs code in the main process. The first argument is always the `electron` module:

        ```js example.spec.js
        import { test } from '@playwright/test'
        import { _electron as electron } from '@playwright/test'

        test('check isPackaged', async () => {
          const electronApp = await electron.launch({ args: ['.'] })

          const isPackaged = await electronApp.evaluate(async ({ app }) => {
            return app.isPackaged
          })

          console.log(isPackaged) // false in development mode
          await electronApp.close()
        })
        ```
      </Step>

      <Step title="Write a full test with assertions and screenshots">
        ```js example.spec.js
        import { test, expect, _electron as electron } from '@playwright/test'

        test('example test', async () => {
          const electronApp = await electron.launch({ args: ['.'] })

          const isPackaged = await electronApp.evaluate(async ({ app }) => {
            return app.isPackaged
          })

          expect(isPackaged).toBe(false)

          // Get the first BrowserWindow as a Playwright Page
          const window = await electronApp.firstWindow()
          await window.screenshot({ path: 'intro.png' })

          await electronApp.close()
        })
        ```
      </Step>

      <Step title="Run your tests">
        ```sh
        npx playwright test
        ```

        Playwright matches any file with `.test.` or `.spec.` in the name. TypeScript is supported out of the box.
      </Step>
    </Steps>
  </Tab>
</Tabs>

## Custom test driver

If you need lower overhead or want to expose custom RPC methods to your test suite, you can build a driver using Node.js's built-in IPC over stdio.

### Driver implementation

```js testDriver.js
const electronPath = require('electron')
const childProcess = require('node:child_process')

class TestDriver {
  constructor({ path, args, env }) {
    this.rpcCalls = []

    env.APP_TEST_DRIVER = 1 // signal to the app to listen for messages
    this.process = childProcess.spawn(path, args, {
      stdio: ['inherit', 'inherit', 'inherit', 'ipc'],
      env
    })

    // match responses to pending RPC calls
    this.process.on('message', (message) => {
      const call = this.rpcCalls[message.msgId]
      if (!call) return
      this.rpcCalls[message.msgId] = null
      if (message.reject) call.reject(message.reject)
      else call.resolve(message.resolve)
    })

    this.isReady = this.rpc('isReady').catch((err) => {
      console.error('Application failed to start', err)
      this.stop()
      process.exit(1)
    })
  }

  async rpc(cmd, ...args) {
    const msgId = this.rpcCalls.length
    this.process.send({ msgId, cmd, args })
    return new Promise((resolve, reject) => this.rpcCalls.push({ resolve, reject }))
  }

  stop() {
    this.process.kill()
  }
}

module.exports = { TestDriver }
```

### App-side message handler

In your Electron app's `main.js`, listen for test driver messages when the `APP_TEST_DRIVER` environment variable is set:

```js main.js
const METHODS = {
  isReady() {
    // perform any setup needed before tests run
    return true
  }
  // add your own RPC-able methods here
}

const onMessage = async ({ msgId, cmd, args }) => {
  let method = METHODS[cmd]
  if (!method) method = () => new Error('Invalid method: ' + cmd)
  try {
    const resolve = await method(...args)
    process.send({ msgId, resolve })
  } catch (err) {
    process.send({ msgId, reject: { message: err.message, stack: err.stack, name: err.name } })
  }
}

if (process.env.APP_TEST_DRIVER) {
  process.on('message', onMessage)
}
```

### Using the driver in a test suite

```js test.js
const electronPath = require('electron')
const test = require('ava')
const { TestDriver } = require('./testDriver')

const app = new TestDriver({
  path: electronPath,
  args: ['./app'],
  env: { NODE_ENV: 'test' }
})

test.before(async () => {
  await app.isReady
})

test.after.always('cleanup', async () => {
  await app.stop()
})
```

<Note>
  The custom driver approach works with any test runner: AVA, Mocha, Jest, and others. The RPC layer is framework-agnostic.
</Note>

## Headless CI testing

Electron is based on Chromium and requires a display driver to launch. On headless Linux CI systems (Travis CI, CircleCI, Jenkins), you need a virtual display.

<Steps>
  <Step title="Install Xvfb">
    Xvfb is a virtual framebuffer that implements the X11 display server protocol entirely in memory. Install it with your system package manager:

    ```sh
    # Debian/Ubuntu
    sudo apt-get install -y xvfb
    ```
  </Step>

  <Step title="Start Xvfb and export DISPLAY">
    Chromium automatically reads the `$DISPLAY` environment variable. Start Xvfb and point `$DISPLAY` at it before running your tests:

    ```sh
    Xvfb :99 -screen 0 1024x768x24 &
    export DISPLAY=':99.0'
    npm test
    ```
  </Step>

  <Step title="Use xvfb-maybe for cross-platform scripts">
    `xvfb-maybe` automatically wraps your command with Xvfb on Linux headless environments and does nothing on Windows or macOS:

    ```sh
    npm install --save-dev xvfb-maybe
    ```

    ```sh
    # In your test script
    xvfb-maybe electron-mocha ./test/*.js
    ```
  </Step>
</Steps>

### CI platform notes

<AccordionGroup>
  <Accordion title="Travis CI">
    Configure Xvfb in your `.travis.yml`. See the [Travis CI documentation on headless browsers](https://docs.travis-ci.com/user/gui-and-headless-browsers/#using-xvfb-to-run-tests-that-require-a-gui) for the full setup.
  </Accordion>

  <Accordion title="CircleCI">
    CircleCI pre-configures Xvfb and sets `$DISPLAY` automatically. No additional configuration is needed.
  </Accordion>

  <Accordion title="Jenkins">
    Install the [Xvfb plugin for Jenkins](https://wiki.jenkins-ci.org/display/JENKINS/Xvfb+Plugin) to manage virtual display setup as part of your build configuration.
  </Accordion>

  <Accordion title="AppVeyor">
    AppVeyor runs on Windows, where a display driver is always available. Electron, Chromium, and Selenium all work out of the box with no additional configuration.
  </Accordion>
</AccordionGroup>
